🐛 버그 #1 — 앱 전체 크래시 (최우선)

CRITICAL
RightPanel ChartPanel: newsItems.slice() TypeError → React 앱 전체 언마운트
파일
frontend/src/components/RightPanel.jsx — ChartPanel 함수
증상
보유 탭에서 종목 클릭 시 차트 탭으로 이동하면 앱 전체가 즉시 흰 화면 / 무응답
근본 원인
getNews()는 { news: [...], recs: [] } 객체를 반환하는데,
ChartPanel은 이를 배열로 착각해 newsItems.slice(0, 5) 호출 →
TypeError: newsItems.slice is not a function → Error Boundary 없어 루트까지 전파 → 앱 전체 크래시
왜 이전에 안 보였나?
이전에는 ChartPanel이 뉴스를 받기 전에 앱이 다른 이유로 느려 증상이 덜 명확했음. 뉴스 데이터 로딩 완료 직후 렌더에서 크래시 발생 (비동기 타이밍 의존) 
// ❌ 버그 코드
const { data: newsItems = [], ... } = useQuery({ queryFn: () => getNews(ticker) })
newsItems.slice(0, 5)  // getNews()는 {news:[...]} 객체 반환 → slice 없음 → TypeError

// ✅ 수정 코드
const { data: newsData, ... } = useQuery({ queryFn: () => getNews(ticker) })
const newsItems = newsData?.news || []   // 객체에서 배열 추출 후 사용

🐛 버그 #2 — 배치 가격 500 에러

HIGH
/api/prices: concurrent.futures.TimeoutError 미처리 → 500 Internal Server Error
파일
backend/main.py — _batch_prices() 함수
증상
51개 티커 배치 가격 요청 시 15초 후 500 에러 반환. 보유 탭 전체 가격 로딩 실패
근본 원인
as_completed(timeout=15)이 15초 후 TimeoutError를 raise하는데 이를 catch하는 코드 없음 → FastAPI가 500 반환.
내부 try/except는 fut.result()만 catch하고 as_completed 자체 예외는 무시
영향 범위
보유 탭 진입 시 (51개 티커 cold cache 경우) 항상 500 에러 발생
# ❌ 버그 코드
for fut in _cf.as_completed(all_futs, timeout=15):  # TimeoutError 발생 시 uncaught
    try:
        ticker, p = fut.result(timeout=0)
    except Exception:
        pass  # fut.result() 예외만 catch, as_completed 예외는 전파됨

# ✅ 수정 코드
try:
    for fut in _cf.as_completed(all_futs, timeout=15):
        try:
            ticker, p = fut.result(timeout=0)
            if p: result[ticker] = p
        except Exception:
            pass
except _cf.TimeoutError:
    pass  # 타임아웃 시 수집된 부분 결과만 반환 (500 에러 방지)

🐛 버그 #4 — 백엔드 yfinance 무한 대기 (이전 세션 발견)

HIGH

영향 함수
_market_data, _stock_full, _kr_history, _stock_news, _stock_peers, _sector_stocks_us, _sp500_heatmap, portfolio_metrics, portfolio_strategy
증상

근본 원인
yf.Ticker().info, .history(), yf.download() 등은 내부 timeout 없음. 네트워크 지연 또는 Yahoo Finance 429 응답 시 무한 대기. 40개 Starlette 스레드 고갈 → 신규 요청 큐 → 전체 앱 응답 불가
수정 내용
모든 yfinance 호출을 ThreadPoolExecutor + timeout으로 래핑. 순차 루프 → 병렬 처리. yf.download() 제거 후 직접 Yahoo v8 HTTP API 호출로 교체. KR 종목: Naver Finance 스크래핑 fast-fail 추가

🔍 근본 원인 분석 (RCA)

주요 원인 #1 — API 계약 불일치
getNews() API 함수가 { news: [...], recs: [] } 객체를 반환하는데, ChartPanel 컴포넌트는 이를 배열로 가정. 타입 문서화 부재 + TypeScript 미사용으로 컴파일 타임 감지 불가. 런타임에만 발생하는 타입 오류가 가장 심각한 버그로 이어짐.
주요 원인 #2 — Error Boundary 부재
React 컴포넌트 렌더 중 발생한 예외가 Error Boundary에 잡히지 않고 앱 루트까지 전파. 결과적으로 화면 전체가 언마운트되어 사용자에게는 "먹통"으로 보임.
주요 원인 #3 — 외부 의존성 timeout 미설정
yfinance 라이브러리의 내부 HTTP 요청에 timeout이 없어, Yahoo Finance 서버 응답 지연 시 Python 스레드가 무한 대기. 동시 다수 요청 시 스레드 풀 고갈로 서버 전체 무응답.

🛡️ 재발 방지 조치

  • API 반환 타입 일관성 — 모든 news API는 { news: [] } 형식 유지. 배열 직접 반환 금지
  • 방어적 배열 접근 — API 데이터 사용 시 항상 data?.news || [] 패턴 사용
  • Error Boundary 추가 고려 — 탭별 Error Boundary로 한 탭 오류가 전체 앱에 영향 주지 않도록
  • yfinance 직접 호출 금지 — 모든 외부 API는 ThreadPoolExecutor(timeout=N)으로 래핑
  • TimeoutError 항상 catch — as_completed(timeout) 사용 시 반드시 except TimeoutError 포함
  • retry 정책 — 차트 등 핵심 쿼리는 retry: 1, retryDelay: 2000으로 일시적 장애 자동 복구
  • 배포 후 대기 — 서버 재시작 후 최소 5초 대기 후 브라우저 새로고침
